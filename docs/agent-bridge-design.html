1. What Pipecat's LLMService Must Do

Architect

Understanding the obligations of an LLMService is the prerequisite for designing a drop-in replacement. The obligations fall into two categories: frames you must receive and act on, and frames you must emit in the right sequence.

Incoming Frames

FrameRequired Action
LLMContextFramePrimary trigger. Contains full conversation context (messages + tools). Run inference, emit response frames.
LLMRunFrameDirect "go run now" signal. Re-run inference with current context.
StartFrameInitialize: create WebSocket connection, start sequential runner task if needed.
EndFrame / CancelFrameDisconnect WebSocket, cancel in-flight tasks.
InterruptionFrameCritical for voice: Cancel all sync function calls and their tasks. Must also abort the remote stream.
LLMUpdateSettingsFrameUpdate runtime settings (temperature, max_tokens, system prompt).
LLMContextSummaryRequestFrameRun out-of-band summary inference, emit LLMContextSummaryResultFrame.

Required Frame Emission Sequence (One Turn)

Receive: LLMContextFrame
   │
   ▼
Emit:   LLMFullResponseStartFrame
   │
   ├── Emit: LLMTextFrame (chunk 1)
   ├── Emit: LLMTextFrame (chunk 2)   ← each text token/word
   │   ...
   │
   │   [if LLM wants to call a tool]
   ├── Emit: FunctionCallsStartedFrame
   ├── Emit: FunctionCallInProgressFrame   ← per tool call
   │        (handler registered in pipecat executes)
   ├── Emit: FunctionCallResultFrame         ← after handler returns
   │        (LLM re-invoked automatically by aggregator)
   │
   ▼
Emit:   LLMFullResponseEndFrame
The Unwritten Protocol

Every LLM service implementation in pipecat wraps the entire inference in LLMFullResponseStartFrame / LLMFullResponseEndFrame. The downstream TTS aggregator depends on this pair to know when to commit audio. If you forget LLMFullResponseEndFrame, the TTS will never speak.

Text chunks must be pushed via self._push_llm_text(text) (not directly via push_frame) so that the turn-completion mixin and skip-TTS logic are correctly applied.

Tool Call Ownership: The Biggest Design Decision

When a remote agent calls a tool internally (e.g., database lookup), pipecat never sees it — the agent just sends back text. But some tools must be executed by pipecat: hangup, transfer, TTS voice change. This creates three modes:

ModeWho runs toolsComplexityUse When
OpaqueRemote agent onlyLowAgent handles all business logic; pipecat only speaks
TransparentPipecat onlyHighFull pipecat tool ecosystem reuse; thin agent
Hybrid (recommended)Agent handles business logic; pipecat handles voice-control toolsMediumMost real-world voice agents

2. Agent Framework Session Abstractions

Architect

Each framework models conversation state differently. Here's a precise map of what each stores and how to bridge to pipecat's LLMContext.

Pipecat LLMContext

LLMContext is a mutable, in-process object rebuilt on every turn from the aggregator. It contains the full conversation history in OpenAI message format:

# Messages are OpenAI ChatCompletionMessageParam format
messages = [
  {"role": "system", "content": "You are a helpful voice assistant"},
  {"role": "user", "content": "Book me a flight to Tokyo"},
  {"role": "assistant", "content": None,
   "tool_calls": [{"id": "call_123", "function": {"name": "search_flights", "arguments": "{}"}}]},
  {"role": "tool", "tool_call_id": "call_123", "content": "[{flight results}]"},
  {"role": "assistant", "content": "I found 3 flights..."},
]

Google ADK Session

ADK stores conversation history as an append-only list of Event objects. Each event uses Google's Content/Part type (same as Gemini API):

session.events = [
  Event(author="user",  content=Content(parts=[Part(text="Book me a flight to Tokyo")])),
  Event(author="model", content=Content(parts=[Part(function_call=FunctionCall(
      name="search_flights", args={}, id="call_123"))])),
  Event(author="search_flights", content=Content(parts=[Part(function_response=FunctionResponse(
      name="search_flights", response={"results": [...]}, id="call_123"))])),
  Event(author="model", content=Content(parts=[Part(text="I found 3 flights...")])),
]
ADK History Translation Problem

The ADK event log includes internal tool calls made by sub-agents, routing decisions, and partial streaming events — none of which appear in pipecat's context. If you reconstruct ADK session state from pipecat messages, you lose this metadata. The recommended approach is to treat the ADK server as authoritative for history and only send it the latest user message per turn.

OpenAI Agents SDK

The SDK tracks conversation in one of two ways:

  • Client-managed: A Session object (protocol) stores list[TResponseInputItem] — the same OpenAI Responses API format, close to pipecat's format.
  • Server-managed: A conversation_id / previous_response_id chain maintained by OpenAI's servers.

For bridge purposes, the client-managed path is easiest: send the full history each time as the input parameter to Runner.run().

LangChain / CrewAI / Others

Most other frameworks accept conversation history as a list of HumanMessage / AIMessage / ToolMessage objects (LangChain) or as plain dicts (CrewAI). These map directly from pipecat's OpenAI-format messages with a simple conversion layer.

Stateless Bridge (Recommended)

Send full conversation history with every request. Server is stateless. Works with all frameworks. Scales horizontally. Slightly more bandwidth.

  • ✅ No session affinity needed
  • ✅ Simple server restart recovery
  • ✅ Works with all frameworks
  • ⚠️ O(n) context per turn

Stateful Bridge (Performance)

Server maintains session state. Pipecat sends only the new user message each turn. Lower bandwidth, but requires session stickiness.

  • ✅ Lower bandwidth
  • ✅ Agent history is authoritative
  • ❌ Session affinity required
  • ❌ Session recovery on crash

The pipecat pipeline is completely standard — STT → LLM → TTS. The only new component is AgentBridgeLLMService, which replaces the direct LLM API call with a WebSocket hop to the customer's agent server.

The agent server is an HTTP/WebSocket endpoint that wraps the customer's chosen framework. VoqalCloud provides adapter libraries for each supported framework (ADK, OpenAI Agents SDK, etc.) to make this trivial to deploy.

4. The Agent Bridge Protocol (Wire Format)

Architect

The protocol runs over a persistent WebSocket connection. One connection per pipecat pipeline session. Each LLM turn is one request-response exchange on that connection.

Connection Lifecycle

# Connect at pipeline start
wss://agent-server/v1/agent?session_id={uuid}&pipeline_id={id}

# Handshake
S→C: {"type": "ready", "session_id": "...", "capabilities": ["interrupt", "tools"]}

# Normal flow: one run per turn
C→S: {"type": "run", ...}
S→C: {"type": "stream_start"}
S→C: {"type": "text_delta", ...}  (repeating)
S→C: {"type": "stream_end"}

# Interrupt: pipecat user spoke mid-stream
C→S: {"type": "interrupt"}
S→C: {"type": "interrupted"}  (server confirms cancellation)

# Disconnect at pipeline end
WebSocket close frame

Request Message: run

{
  "type": "run",
  "turn_id": "turn-uuid-123",       // for correlation

  // Strategy A (stateless): send full history every time
  "messages": [                        // OpenAI-format message array
    {"role": "system", "content": "..."},
    {"role": "user", "content": "Book me a flight"},
    {"role": "assistant", "content": "Sure! Where to?"},
    {"role": "user", "content": "Tokyo, next Tuesday"}
  ],

  // Optional: override agent settings
  "settings": {
    "system_instruction": "...",     // if set, overrides agent's default
    "max_tokens": 512,
    "temperature": 0.7
  },

  // Pipecat-side tools the agent can invoke (hybrid mode)
  "pipecat_tools": [                  // OpenAI function schema format
    {"name": "hangup", "description": "...", "parameters": {}},
    {"name": "transfer", "description": "...", "parameters": {"destination": "..."}}
  ]
}

Response Event Stream

// 1. Stream begins
{"type": "stream_start", "turn_id": "turn-uuid-123"}

// 2. Text chunks (emit one per LLM token or word boundary)
{"type": "text_delta", "text": "I found"}
{"type": "text_delta", "text": " three flights"}

// 3a. [Hybrid mode] Agent wants pipecat to execute a tool
{"type": "tool_call",
 "id": "call_456",
 "name": "transfer",
 "arguments": {"destination": "sales-queue"}}

// 3b. [Hybrid mode] Pipecat sends tool result back
// C→S: {"type": "tool_result", "id": "call_456", "result": {"status": "ok"}}

// 4. End of stream
{"type": "stream_end",
 "turn_id": "turn-uuid-123",
 "usage": {"input_tokens": 312, "output_tokens": 47}}

// Error path
{"type": "error", "code": "agent_timeout", "message": "...", "fatal": false}

Interrupt Protocol

// Pipecat detects user speech → sends interrupt immediately
C→S: {"type": "interrupt", "turn_id": "turn-uuid-123"}

// Server MUST:
// 1. Stop generating immediately
// 2. Cancel any in-flight tool calls
// 3. NOT append the partial response to session history
// 4. Confirm cancellation
S→C: {"type": "interrupted", "turn_id": "turn-uuid-123"}

// Server is now idle, waiting for next "run"
Critical: Interrupt Must Be Fast

From when pipecat sends {"type":"interrupt"} to when TTS audio stops, the latency budget is ~100ms. The server must cancel its LLM stream immediately (not wait for the current token). Google ADK and OpenAI Agents SDK both support async cancellation but you must wire it correctly — cancelling the Python asyncio.Task running runner.run_async() is sufficient for most frameworks.

5. Context / History Translation

Architect

This is the most subtle part of the design. Pipecat's LLMContext and each framework's session format overlap significantly but not perfectly.

Translation Map

OpenAI Format (Pipecat)Google ADKOAI Agents SDKLangChain
{"role":"user","content":"Hi"} Event(author="user", content=Content(parts=[Part(text="Hi")])) {"role":"user","content":[{"type":"input_text","text":"Hi"}]} HumanMessage(content="Hi")
{"role":"assistant","content":"Hello!"} Event(author="model", content=Content(parts=[Part(text="Hello!")])) {"role":"assistant","content":[{"type":"output_text","text":"Hello!"}]} AIMessage(content="Hello!")
{"role":"assistant","tool_calls":[{"id":"c1","function":{"name":"f","arguments":"{}"}}]} Event(author="model", content=Content(parts=[Part(function_call=FunctionCall(...))])) {"type":"function_call","id":"c1","name":"f","arguments":"{}"} AIMessage(tool_calls=[ToolCall(...)])
{"role":"tool","tool_call_id":"c1","content":"result"} Event(author="f", content=Content(parts=[Part(function_response=FunctionResponse(...))])) {"type":"function_call_output","call_id":"c1","output":"result"} ToolMessage(tool_call_id="c1", content="result")

The "Last Message Only" Strategy (Stateful Server)

If the remote agent server maintains session state (e.g., ADK session or OpenAI conversation_id), you should only send the new user message, not the full history. The server tracks the rest.

This requires the bridge to:

  1. Track which messages have already been sent (by count or hash)
  2. Extract only the delta (new messages since last turn)
  3. Send just the delta to the server
# In AgentBridgeLLMService (stateful mode)
def _get_new_messages(self, context: LLMContext) -> list[dict]:
    messages = context.get_messages()
    # Track how many we've sent; only send the tail
    new_from = self._sent_message_count
    self._sent_message_count = len(messages)
    return messages[new_from:]
Context Drift Risk

If pipecat's context and the server's session get out of sync (e.g., after an interruption that partially appended a response), you will get hallucinated tool call references or nonsensical follow-ups. The safest recovery is a {"type":"reset","messages":[...]} message that lets the server rebuild from the full context. Build this as a fallback from day one.

System Instruction Precedence

Both pipecat (via LLMSettings.system_instruction) and the agent framework (the agent's own instructions/persona) may define a system prompt. Establish a clear precedence rule:

  1. Agent's persona (defined in framework config) — foundational identity
  2. VoqalCloud base instructions — turn completion markers if enabled, voice output constraints
  3. Runtime override (from LLMUpdateSettingsFrame) — per-call overrides, dynamic context

The bridge protocol's settings.system_instruction field should only carry the runtime override; the agent server merges it with the agent's own persona.

6. Implementing AgentBridgeLLMService

Architect

AgentBridgeLLMService extends WebsocketLLMService. The heavy lifting (reconnection, connection lifecycle, concurrent inference management) is already done. We implement three things: connect, send, receive.

class AgentBridgeLLMService(WebsocketLLMService):
    """Bridges pipecat to a remote Agent Bridge Protocol server."""

    def __init__(self, *, url: str, session_id: str, api_key: str | None = None, **kwargs):
        super().__init__(**kwargs)
        self._url = url
        self._session_id = session_id
        self._api_key = api_key
        self._current_turn_id: str | None = None
        self._interrupted = asyncio.Event()

    # ── Connection ────────────────────────────────────────────────

    async def _connect_websocket(self):
        headers = {"Authorization": f"Bearer {self._api_key}"} if self._api_key else {}
        self._websocket = await websockets.connect(
            f"{self._url}?session_id={self._session_id}", extra_headers=headers
        )
        ready = await self._ws_recv()
        assert ready["type"] == "ready"

    async def _disconnect_websocket(self):
        if self._websocket:
            await self._websocket.close()

    # ── Main inference ────────────────────────────────────────────

    async def process_frame(self, frame: Frame, direction: FrameDirection):
        await super().process_frame(frame, direction)
        if isinstance(frame, LLMContextFrame):
            try:
                await self.push_frame(LLMFullResponseStartFrame())
                await self._process_context(frame.context)
            except Exception as e:
                await self.push_error(str(e), exception=e, fatal=False)
            finally:
                await self.push_frame(LLMFullResponseEndFrame())
        else:
            await self.push_frame(frame, direction)

    async def _process_context(self, context: LLMContext):
        self._current_turn_id = str(uuid.uuid4())
        self._interrupted.clear()

        # Build and send the run request
        request = {
            "type": "run",
            "turn_id": self._current_turn_id,
            "messages": context.get_messages(),
            "settings": {"system_instruction": self._settings.system_instruction},
            "pipecat_tools": self._get_pipecat_tool_schemas(),
        }
        await self._ws_send(request)

        # Consume the event stream
        pending_tool_calls: list[FunctionCallFromLLM] = []
        while True:
            event = await self._ws_recv()
            etype = event["type"]

            if etype == "text_delta":
                await self._push_llm_text(event["text"])

            elif etype == "tool_call":
                pending_tool_calls.append(FunctionCallFromLLM(
                    context=context,
                    tool_call_id=event["id"],
                    function_name=event["name"],
                    arguments=json.loads(event.get("arguments", "{}")),
                ))

            elif etype == "stream_end":
                if pending_tool_calls:
                    await self.run_function_calls(pending_tool_calls)
                break

            elif etype == "error":
                await self.push_error(event["message"], fatal=event.get("fatal", False))
                break

    # ── Interruption ──────────────────────────────────────────────

    async def _handle_interruptions(self, frame: InterruptionFrame):
        await super()._handle_interruptions(frame)
        if self._current_turn_id:
            await self._ws_send({
                "type": "interrupt",
                "turn_id": self._current_turn_id
            })
            # Don't wait for "interrupted" ack — fire and forget

7. Remote Agent Server Adapters

Developer

VoqalCloud provides adapter libraries that turn any agent framework into an Agent Bridge Protocol-compliant WebSocket server. Below is the structure for each major framework.

Google ADK Adapter

from fastapi import FastAPI, WebSocket
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService
from voqalcloud.bridges.adk import AgentBridgeADKAdapter

app = FastAPI()
session_service = InMemorySessionService()

# Your ADK agent definition
runner = Runner(agent=your_root_agent, session_service=session_service, ...)

adapter = AgentBridgeADKAdapter(runner=runner, session_service=session_service)

@app.websocket("/v1/agent")
async def agent_ws(ws: WebSocket, session_id: str):
    await adapter.handle(ws, session_id)

# AgentBridgeADKAdapter internally does:
# 1. Receive "run" message → extract messages (last user message for stateful, or full history)
# 2. Convert messages to ADK Content/Part format
# 3. Call runner.run_async(user_id, session_id, new_message, run_config=RunConfig(streaming_mode=SSE))
# 4. For each Event from the generator:
#    - partial text event → send {"type":"text_delta","text":...}
#    - function_call event → if in pipecat_tools: send {"type":"tool_call",...}
#                          → else: execute internally, inject result, continue
# 5. End of generator → send {"type":"stream_end"}
# 6. On "interrupt" → cancel the asyncio.Task running run_async

OpenAI Agents SDK Adapter

from agents import Agent, Runner
from voqalcloud.bridges.openai_agents import AgentBridgeOAIAdapter

# Your agent definition
agent = Agent(name="Support Bot", instructions="...", tools=[...])
adapter = AgentBridgeOAIAdapter(agent=agent)

@app.websocket("/v1/agent")
async def agent_ws(ws: WebSocket, session_id: str):
    await adapter.handle(ws, session_id)

# AgentBridgeOAIAdapter internally does:
# 1. Receive "run" → convert messages to TResponseInputItem format
# 2. Call Runner.run_streamed(agent, input=messages)
# 3. async for event in streamed_result.stream_events():
#    - RawResponsesStreamEvent (text delta) → {"type":"text_delta","text":...}

#                                             → else: allow SDK to handle it
# 4. stream_events() exhausted → {"type":"stream_end"}
# 5. On "interrupt" → cancel the streaming task

8. Latency Design

ArchitectDeveloper

Voice agents have an unforgiving latency budget. The user expects a response within ~1.5s of finishing speaking. Every ms at the agent bridge seam must be budgeted carefully.

Latency Budget (Realistic Estimates)

STT (Deepgram Nova-3 streaming)
~50ms
Pipecat frame processing
~5ms
WebSocket hop (same region)
~10ms
Agent server processing (before streaming)
~50ms
LLM Time-to-First-Token
~400-700ms
First text_delta → TTS → first audio
~200-400ms

Total before first audio: ~720ms–1200ms, dominated by LLM TTFT. Adding the bridge WebSocket hop adds only ~10-20ms. This is acceptable.

Design Rules for Low Latency

Rule 1: Pre-connect the WebSocket

Connect the WebSocket on StartFrame (pipeline startup), not on the first LLM request. The pipecat pipeline starts before any user speaks. Use the warm-up time.

Rule 2: Stream Tokens Immediately, Don't Buffer

The server must emit text_delta events for every token, not buffer a complete sentence. TTS (Cartesia, ElevenLabs) can begin generating audio from 10-15 characters. Every extra character buffered on the server side delays first audio by ~5ms.

Rule 3: Co-locate the Agent Server with the LLM Provider

The biggest latency source is LLM TTFT. If you're using OpenAI's API, run the agent server in Azure East US (OpenAI's primary region). If using Gemini, use GCP us-central1. The bridge WebSocket hop is small; the LLM API hop is large.

Rule 4: Keep the Agent Server Warm

Cold start on a serverless Python container adds 500ms-2s. Use long-running containers (Cloud Run with min-instances=1, or ECS, or Kubernetes). The pipecat server also stays warm — the WebSocket connection carries the lifecycle signal.

Rule 5: Avoid Tool-Call Round Trips in the Hot Path

Each tool call in hybrid mode requires a round trip: server→pipecat→server. This is 20-40ms RTT. Agent tools that must return before the LLM continues (e.g., "lookup customer name") should be executed on the agent server side, not delegated to pipecat. Only delegate tools that need pipecat state (hangup, transfer, audio control).

Rule 6: Use Interruption to Cancel Tool Calls Too

If the agent is mid-tool-call when an interruption arrives, the server must cancel the tool call execution (not just the LLM stream). For long-running async tools, implement cancellation via asyncio.Task.cancel(). Don't wait for the tool to finish — cancel immediately.

Latency Target

With same-region co-location and proper streaming: the bridge adds <20ms total. Time-to-first-audio remains dominated by LLM TTFT (400-700ms for most models), which is identical to what you'd get with direct LLM integration. The bridge is not a meaningful latency disadvantage.

9. Developer Constraints & Limitations

DeveloperImportant

This section is written for developers integrating their agents with VoqalCloud. These are real constraints you must design around, not suggestions.

  • Hard
    Your agent must support async streaming. Synchronous agents (those that return a complete response, not a stream) will produce poor voice UX — the user hears nothing until the full response is ready. All supported frameworks (ADK, OpenAI Agents, LangChain) have streaming APIs. You must use them. A blocking agent.run() call behind this WebSocket will cause noticeable pauses.
  • Hard
    Your agent must handle abrupt cancellation. When a user interrupts the bot mid-sentence, your agent's current invocation is cancelled. Any state mutations your agent has already made (database writes, API calls) will NOT be rolled back — they happened before the cancel signal. If your agent writes state mid-response (e.g., "I've sent the email" then sends the email in a tool call), the action may have already executed when the response is cancelled. Design for idempotency or defer side effects to the end of a turn.
  • Hard
    Context is owned by pipecat, not your agent. Pipecat's context aggregator is the source of truth for conversation history. If your agent framework maintains its own session state (ADK's session.events), it may diverge from pipecat's view after interruptions or errors. You must either (a) use stateless mode and reconstruct from pipecat's context each turn, or (b) implement a reconciliation protocol. Option (a) is strongly recommended for most use cases.
  • Medium
    Voice-appropriate response length. LLM responses appropriate for text (bullet lists, code blocks, long paragraphs) are wrong for voice. Your agent's system prompt must constrain response length and format. VoqalCloud can inject base voice instructions, but your agent's persona instructions take precedence if they conflict. Design your agent's system prompt for spoken output from day one.
  • Medium
    Tool call latency compounds. Each tool call your agent makes (regardless of whether it's handled on the agent side or by pipecat) adds LLM inference latency for the follow-up response. A chain of 3 tool calls can add 1-2 seconds before the user hears a response. Prefer batched lookups over sequential tool calls for time-sensitive paths.
  • Medium
    Multi-agent handoffs are transparent to pipecat. If your agent uses sub-agents or handoffs (ADK's transfer_to_agent, OAI Agents' handoffs), pipecat doesn't know which sub-agent is speaking. Voice indicators (tone, pacing) are consistent — there's no built-in mechanism to switch TTS voice when a handoff occurs. If you need voice persona switching on handoff, you must emit a custom tool call to pipecat (hybrid mode).
  • Medium
    Pipecat function handlers run in the pipeline, not the agent server. In hybrid mode, when your agent emits a tool_call for pipecat to execute, the handler registered via llm.register_function() runs inside the pipecat process. This handler has access to pipecat state (transport, context) but NOT to your agent server's state or databases. Keep pipecat-side handlers focused on voice control (hangup, transfer, audio) and let the agent handle business logic.
  • Soft
    Debugging is harder. The full trace of a voice turn now spans two processes: pipecat and the agent server. Without distributed tracing (correlation via turn_id), diagnosing latency or correctness issues is painful. Emit turn_id in all log lines on both sides from day one.
  • Soft
    The bridge adds a network hop, which affects local development. In production the agent server is co-located (low RTT). Locally, running the agent server as a separate process on the same machine adds negligible latency, but adds operational complexity. VoqalCloud should provide a voqal dev CLI command that starts both the pipecat pipeline and the agent server together.

What the Bridge Does NOT Limit

These capabilities are fully preserved through the bridge:

  • Any LLM provider (OpenAI, Gemini, Claude, Llama, Mistral) — the agent server talks to whatever LLM you choose
  • Any agent pattern (ReAct, plan-and-execute, multi-agent, RAG)
  • Any memory system (vector DB, Redis, SQL) — lives in the agent server
  • Any tool ecosystem — agent-side tools are invisible to pipecat
  • Pipecat's full STT/TTS/transport ecosystem
  • Pipecat's interruption handling, VAD, turn completion detection
  • All pipecat observability (metrics, tracing, RTVI)

Key Invariants

  1. LLMFullResponseStartFrame and LLMFullResponseEndFrame must be emitted as a pair, even on error. Use try/finally.
  2. Text must be pushed via self._push_llm_text(text), not push_frame(LLMTextFrame(text)) directly, to respect turn-completion and skip-TTS settings.
  3. Function calls must go through self.run_function_calls(calls). Do not emit FunctionCallInProgressFrame or FunctionCallResultFrame manually.
  4. All downstream frame pushes must use await self.push_frame(frame). Never push from a thread; always from an async context.
  5. The InterruptionFrame must be passed through downstream (via await super().process_frame(frame, direction)) so downstream processors (TTS, output transport) also get the signal.
